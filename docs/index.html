Why meldlane-tasks

One contract, swappable backends, idempotent by construction.

🔀

Swap the backend, not the call site

Start with a markdown file, move to SQLite, sync to Plane later — the calling code stays `await sink.push(task)`.

♻️

Idempotent by construction

PlaneSink derives a stable external_id from your task id. Pushing the same task twice updates it — verified live: 13 tasks pushed twice, still 13 issues.

⚖️

Honest about the gaps

Plane expects assignee to be a real user account — this library doesn't fake agent-as-assignee support. Assignee, story points and source go into the description as text.

📦

Library, not a service

No server, no config file, no daemon. Just async functions and a Pydantic model.

Sinks

Three today. The TaskSink interface leaves room for more without breaking existing ones.

sinkbackendnotes
MarkdownSinka markdown file Writes/updates a table row. Human-readable, not a full round-trip.
SqliteSinklocal SQLite file Full round-trip — every Task field survives push() → list().
PlaneSinkPlane REST API Idempotent via external_id/external_source. list() filters to issues this library created.

No Jira / YouTrack adapters

⚠ And that's on purpose

Writing an integration against an API you can't test live against a real instance produces code nobody should trust. Rather than ship an untested guess, they're simply not here.

The TaskSink interface is stable enough to add them later — contributions with a real instance to test against are welcome.

Install

Python 3.11+. No config file — pass credentials to the sink you use.
